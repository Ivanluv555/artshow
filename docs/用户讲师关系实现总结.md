### 1. 数据库表设计（满足第三范式）

#### 1.1 表结构
**文件：** `database/user_instructor_relationship.sql`

```sql
CREATE TABLE user_instructor_relationship (
    relationship_id BIGINT PRIMARY KEY COMMENT '关系ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    instructor_id BIGINT NOT NULL COMMENT '讲师ID',
    established_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '关系建立时间',
    
    UNIQUE KEY uk_user_instructor (user_id, instructor_id),
    
    CONSTRAINT fk_relationship_user
        FOREIGN KEY (user_id) REFERENCES user(user_id) ON DELETE CASCADE,
    CONSTRAINT fk_relationship_instructor
        FOREIGN KEY (instructor_id) REFERENCES course_instructor(instructor_id) ON DELETE CASCADE,
    
    INDEX idx_user_id (user_id),
    INDEX idx_instructor_id (instructor_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 1.2 第三范式验证 ✅

**1NF（第一范式）：** 所有字段都是原子的
- `relationship_id`: 单一主键
- `user_id`: 单一用户标识
- `instructor_id`: 单一讲师标识
- `established_at`: 单一时间戳

**2NF（第二范式）：** 非主属性完全依赖于主键
- 主键: `relationship_id`
- 所有非主属性（`user_id`, `instructor_id`, `established_at`）都完全依赖于主键
- 不存在部分依赖

**3NF（第三范式）：** 非主属性之间不存在传递依赖
- `user_id` 不依赖于 `instructor_id`
- `instructor_id` 不依赖于 `user_id`
- `established_at` 不依赖于其他非主属性
- 所有非主属性都直接依赖于主键，不存在传递依赖

#### 1.3 表特性

**唯一约束：**
- `uk_user_instructor (user_id, instructor_id)` - 防止重复创建关系

**外键约束：**
- `fk_relationship_user` - 级联删除（用户删除时自动删除关系）
- `fk_relationship_instructor` - 级联删除（讲师删除时自动删除关系）

**索引优化：**
- `idx_user_id` - 快速查询用户的所有讲师
- `idx_instructor_id` - 快速查询讲师的所有学生

## 🛡️ 安全特性

### 1. 防止重复创建关系

**机制：**
- 数据库层面：唯一约束 `uk_user_instructor (user_id, instructor_id)`
- 应用层面：调用前检查 `existsByUserIdAndInstructorId()`

**好处：**
- 避免数据冗余
- 防止违反唯一约束异常

### 2. 讲师无权主动建立关系

**机制：**
- `createRelationship()` 方法只在内部调用（Service层）
- 没有对外暴露的API接口
- 只能通过课程报名触发

**实现方式：**
- 不创建Controller接口
- Service方法不标记为 `@Public`
- 仅在 `CourseEnrollmentService` 内部调用

### 3. 级联删除保护

**外键约束：**
- 用户删除 → 自动删除所有相关关系
- 讲师删除 → 自动删除所有相关关系

**好处：**
- 防止孤儿数据
- 保持数据一致性

### 4. 事务一致性

**特性：**
- 报名和建立关系在同一事务中
- 任何步骤失败都会回滚
- 保证数据完整性

## 📈 性能考虑

### 查询效率

**索引优化：**
- `idx_user_id`: 查询用户的讲师列表
- `idx_instructor_id`: 查询讲师的学生列表
- 复合唯一索引: 检查关系是否存在

**时间复杂度：**
- 检查关系存在: O(1) - 索引查询
- 查询用户的讲师: O(N) - N为讲师数量
- 统计数量: O(1) - 数据库COUNT优化

### 存储空间

**每条记录：**
- `relationship_id`: 8字节（BIGINT）
- `user_id`: 8字节（BIGINT）
- `instructor_id`: 8字节（BIGINT）
- `established_at`: 8字节（TIMESTAMP）
- **总计**: 约32字节 + 索引开销

**估算：**
- 1000个用户，平均每人3个讲师 = 3000条记录
- 存储空间: 3000 × 32字节 ≈ 96KB
- 非常轻量级

## ⚠️ 注意事项

### 1. 数据库迁移

**对于已有数据：**
- 执行建表SQL: `user_instructor_relationship.sql`
- 为现有的课程报名记录补充关系数据

**补充脚本示例：**
```sql
INSERT INTO user_instructor_relationship (relationship_id, user_id, instructor_id, established_at)
SELECT 
    -- 使用雪花ID生成器或自增ID
    ROW_NUMBER() OVER (ORDER BY uce.user_id, c.instructor_id) as relationship_id,
    uce.user_id,
    c.instructor_id,
    uce.enrolled_at
FROM user_course_enrollment uce
JOIN course c ON uce.course_id = c.course_id
WHERE c.instructor_id IS NOT NULL
GROUP BY uce.user_id, c.instructor_id
ON DUPLICATE KEY UPDATE user_id = user_id; -- 忽略重复记录
```
