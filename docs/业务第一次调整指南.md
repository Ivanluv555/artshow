# 业务第一次调整指南

**项目名称：** Artshow - 艺术展览与在线课程管理系统
**创建日期：** 2026-07-05  
**文档类型：** 业务逻辑分析与调整建议

---

## 📋 文档概述

本文档基于当前代码和数据库结构，分析了系统中存在的业务逻辑不完善之处，并提供调整建议。分析聚焦于业务逻辑层面，不涉及技术实现细节。

**业务决策确认：**

- ✅ 课程模式：混合模式（部分免费、部分付费）
- ✅ 讲师身份：讲师是用户的子集（需要关联user表）
- ✅ 卖家权限：卖家可以自主管理自己的商品
- ✅ 徽章获取：自动触发机制（达成条件自动授予）

## 🔴 严重问题（影响核心业务流程）

### 1. 订单流程严重不完整

**问题描述：**

- 订单创建（`OrderService.addOrder`）只保存订单主表，但没有创建订单项（order_item）
- 缺少"购物车结算→生成订单"的完整流程
- 订单号（order_number）字段未生成，会导致插入失败（UNIQUE约束）
- 没有库存检查和扣减逻辑
- 订单项的价格快照（price_at_purchase）和商品快照（product_name_snapshot, product_image_snapshot）未填充

**业务影响：**

- 用户无法正常下单购买商品
- 订单数据不完整，无法追溯商品信息
- 库存管理失控，可能超卖

**调整建议：**

1. 新增业务方法：`createOrderFromCart(List<Integer> cartItemIds, Integer addressId)`
   - 从购物车项获取商品信息
   - 检查每个商品的库存是否充足
   - 生成唯一订单号（建议格式：`ORD + 时间戳 + 随机数`）
   - 创建订单主表记录
   - 创建所有订单项，填充价格快照和商品快照
   - 扣减商品库存
   - 清空已结算的购物车项
   - 使用 `@Transactional` 保证原子性

2. 新增订单状态流转方法：
   - `cancelOrder(Integer orderId)` - 取消订单，恢复库存
   - `payOrder(Integer orderId)` - 标记订单已支付
   - `shipOrder(Integer orderId)` - 标记订单已发货
   - `completeOrder(Integer orderId)` - 完成订单

---

### 2. 课程付费逻辑缺失

**问题描述：**

- 课程表有 `price` 字段，但缺少 `is_free` 或 `type` 字段区分免费/付费
- 课程报名（`enrollCourse`）直接创建注册记录，没有支付环节
- 付费课程应该通过订单购买，但 order_item 只关联 product，不关联 course

**业务影响：**

- 混合模式无法实现（无法区分免费和付费课程）
- 付费课程可以免费学习，商业模式失效

**调整建议：**

1. 数据库调整：
   - 在 `course` 表增加 `is_free` 字段（BOOLEAN，默认false）
   - 或增加 `course_type` 字段（ENUM: 'free', 'paid'）

2. 业务流程调整：
   - 免费课程：保持现有 `enrollCourse` 直接报名逻辑
   - 付费课程：
     - 方案A：扩展 `order_item` 表，增加可选的 `course_id` 字段（product_id 和 course_id 二选一）
     - 方案B：创建专门的 `course_order` 和 `course_order_item` 表
     - 推荐方案A，统一订单体系

3. 报名检查：
   - `enrollCourse` 前检查：
     - 如果是付费课程，验证用户是否已购买（order_item中存在该课程且订单已支付）
     - 如果未购买，返回错误提示用户先购买

---

### 3. 库存管理完全缺失

**问题描述：**

- `product` 表有 `stock` 字段，但业务代码中完全未使用
- 下单时不检查库存
- 支付后不扣减库存
- 取消订单不恢复库存

**业务影响：**

- 可能出现超卖（库存为0仍可下单）
- 库存数据失真，无法反映真实可售数量

**调整建议：**

1. 在"购物车结算生成订单"流程中：
   - 下单前：检查所有商品 `stock >= quantity`，任一不足则拒绝下单
   - 下单时：执行 `UPDATE product SET stock = stock - ? WHERE product_id = ? AND stock >= ?`
   - 如果更新影响行数为0，说明库存不足或商品不存在，回滚事务

2. 在订单取消流程中：
   - 恢复库存：`UPDATE product SET stock = stock + ? WHERE product_id = ?`
   - 仅当订单状态为 'pending' 或 'paid'（未发货）时允许取消

3. 在商品管理中：
   - 卖家修改库存时，检查是否有待发货订单占用库存

## 🟠 重要问题（影响用户体验和数据安全）

### 4. 权限控制严重不足

**问题描述：**

#### 4.1 帖子（Post）无权限控制

- `PostService` 的 `deletePost`、`updatePost` 方法没有检查当前用户是否是帖子作者
- 任何用户可以删除或修改他人的帖子

#### 4.2 商品（Product）无权限控制

- `ProductService` 的 `updateOneProduct`、`deleteOneProduct` 没有检查当前用户是否是卖家
- 任何用户可以修改或删除他人的商品

#### 4.3 评论（Comment）可能存在同样问题

- 需要确认评论的删除和修改权限

**业务影响：**

- 用户数据可被恶意篡改或删除
- 商业数据（商品信息、价格）可被竞争对手恶意修改
- 用户体验极差，内容安全无法保证

**调整建议：**

1. **Post服务调整：**

   ```java
   deletePost(Integer postId) {
       Post post = 查询帖子;
       if (!post.getUserId().equals(当前用户ID)) {
           throw 无权限异常;
       }
       删除操作;
   }
   ```

   同样逻辑应用于 `updatePost`

2. **Product服务调整：**

   ```java
   updateOneProduct(ProductDTO dto) {
       Product product = 查询商品;
       if (!product.getSellerId().equals(当前用户ID)) {
           throw 无权限异常;
       }
       更新操作;
   }
   ```

   同样逻辑应用于 `deleteOneProduct`

3. **Comment/Like/Collection服务检查：**
   - 确认是否已有权限控制
   - 如无，参照上述模式添加

### 5. 用户密码安全问题

**问题描述：**

- `UserService.login` 方法中使用明文密码比较：`user.getPassword().equals(password)`
- 数据库字段名为 `password_hash`，但可能存储的是明文密码

**业务影响：**

- 用户密码泄露风险极高
- 不符合安全标准，可能违反数据保护法规

**调整建议：**

1. 注册时：使用 BCrypt 或其他加密算法加密密码后存储
2. 登录时：将用户输入的密码加密后与数据库中的密文比对
3. 不要在日志中记录密码相关信息
4. 数据迁移：对现有明文密码进行加密（需要通知用户重置密码或使用迁移脚本）

---

### 6. 讲师与用户关系未建立

**问题描述：**

- `course_instructor` 表是独立的，没有关联 `user` 表
- 根据业务决策，讲师应该是用户的子集，讲师需要登录系统管理自己的课程
- 当前讲师无法登录，也无法管理课程

**业务影响：**

- 讲师无法自助管理课程内容
- 需要管理员人工维护讲师信息，效率低
- 讲师与用户账号脱节，体验差

**调整建议：**

1. **数据库调整：**
   - 在 `course_instructor` 表增加 `user_id` 字段（可空，用于兼容历史数据）
   - 添加外键约束：`FOREIGN KEY (user_id) REFERENCES user(user_id)`
   - 添加唯一索引：`UNIQUE KEY uk_user_id (user_id)`（一个用户只能是一个讲师）

2. **业务流程调整：**
   - 用户申请成为讲师：在 `course_instructor` 表创建记录并关联 user_id
   - 讲师信息（name, title, bio, avatar_url）可以与用户信息同步或独立维护
   - 课程管理权限：只有课程的 instructor_id 对应的讲师用户可以修改课程

3. **课程服务调整：**
   - `updateCourse` 和 `deleteCourse` 需要检查：
     - 当前用户是否是该课程的讲师
     - 或者是管理员

## 🟡 次要问题（影响功能完整性）

### 7. 数据统计字段未维护

**问题描述：**

#### 7.1 课程学生数未更新

- `course` 表的 `student_count` 字段存在
- 但 `enrollCourse` 时未自动 +1
- 退课（如果有）时未 -1

#### 7.2 帖子互动数据需要实时统计

- 帖子的点赞数、评论数、收藏数需要频繁查询
- 当前需要关联查询 `post_like`、`post_comment`、`post_collection` 表
- 性能较差

**业务影响：**

- 课程学生数不准确，影响课程热度展示
- 帖子列表加载缓慢（需要多次关联查询）

**调整建议：**

1. **课程学生数维护：**
   - `enrollCourse` 成功后：`UPDATE course SET student_count = student_count + 1 WHERE course_id = ?`
   - 如果支持退课，退课后 -1

2. **帖子统计数据（二选一）：**
   - **方案A（冗余字段）：**
     - 在 `post` 表增加 `like_count`、`comment_count`、`collection_count` 字段
     - 点赞/评论/收藏时更新对应字段
     - 适合读多写少场景
   - **方案B（实时统计）：**
     - 保持现状，查询时关联统计
     - 添加缓存（Redis）存储统计数据
     - 适合数据一致性要求高的场景

### 8. 默认地址逻辑不完整

**问题描述：**

- `user_address` 表有 `is_default` 字段
- 用户设置新的默认地址时，没有将其他地址的 `is_default` 改为 false
- 可能导致一个用户有多个默认地址

**业务影响：**

- 下单时无法确定使用哪个地址
- 用户体验混乱

**调整建议：**
在地址服务的"设置默认地址"或"新增地址为默认"方法中：

```java
setDefaultAddress(Integer addressId) {
    1. 验证地址属于当前用户
    2. 将该用户的所有地址设为非默认：
       UPDATE user_address SET is_default = 0 WHERE user_id = ?
    3. 将指定地址设为默认：
       UPDATE user_address SET is_default = 1 WHERE address_id = ?
    4. 使用事务保证原子性
}
```

### 9. 课程证书颁发逻辑缺失

**问题描述：**

- `user_course_enrollment` 表有 `certificate_awarded` 和 `award_date` 字段
- 但业务代码中没有颁发证书的逻辑
- 不清楚颁发证书的条件

**业务影响：**

- 证书系统无法使用
- 用户完成课程后没有成就感和激励

**调整建议：**

1. **定义颁发条件（建议）：**
   - 完成课程的所有章节
   - 或达到特定完成比例（如80%）

2. **实现自动颁发：**
   - 在 `completeChapter` 方法中，章节完成后检查：

     ```java
     if (该课程所有章节都已完成) {
         UPDATE user_course_enrollment 
         SET certificate_awarded = 1, award_date = NOW() 
         WHERE enrollment_id = ?
         
         // 可选：触发徽章系统，授予"课程完成"徽章
     }
     ```

3. **提供证书查询接口：**
   - 查询用户已获得的所有证书
   - 生成证书图片或PDF（后续功能）

### 10. 徽章自动触发机制未实现

**问题描述：**

- 根据业务决策，徽章应该在达成条件时自动授予
- 当前只有 `badge` 表定义徽章，`user_badge` 表记录用户徽章
- 但没有任何业务代码检查和授予徽章

**业务影响：**

- 徽章系统形同虚设
- 用户缺少激励机制

**调整建议：**

1. **定义徽章类型和触发条件（示例）：**
   - "新手上路"：注册成功后自动授予
   - "学习达人"：完成3门课程
   - "社交之星"：发布10个帖子
   - "艺术家"：商品被购买10次
   - "课程完成"：完成任意一门课程

2. **实现触发检查：**
   - 在关键业务节点调用徽章检查方法：

     ```java
     void checkAndAwardBadges(Integer userId, String eventType) {
         根据eventType查询对应的徽章规则
         检查用户是否满足条件
         如果满足且未拥有该徽章，则授予
     }
     ```

   - 调用时机：
     - 用户注册后
     - 课程完成后
     - 发帖后（需统计总帖子数）
     - 商品被购买后（需统计销量）

3. **徽章配置化（高级）：**
   - 增加 `badge_rule` 表存储徽章规则
   - 字段：badge_id, condition_type, condition_value
   - 例如：badge_id=3, condition_type='course_completed', condition_value='3'

## 🔵 增强建议（提升用户体验）

### 11. 订单查询功能不足

**问题描述：**

- 用户只能查询单个订单（需要知道orderId）
- 缺少"我的订单列表"功能
- 缺少订单状态筛选、时间范围筛选

**调整建议：**

1. 已有 `findMyOrders()` 方法，应暴露为API接口
2. 扩展查询功能：
   - 按状态筛选：`findMyOrdersByStatus(String status)`
   - 按时间范围：`findMyOrdersByDateRange(Date start, Date end)`
   - 分页查询：使用 Spring Data 的 `Pageable`

---

### 12. 购物车功能不完整

**问题描述：**

- 可以添加、删除、修改单个购物车项
- 但缺少：
  - 批量删除（清空购物车）
  - 批量查询（查看我的购物车）
  - 购物车商品数量统计

**调整建议：**

1. 新增方法：
   - `findMyCart()` - 查询当前用户的购物车
   - `clearMyCart()` - 清空购物车
   - `getCartItemCount()` - 获取购物车商品种类数

2. 返回数据优化：
   - 查询购物车时，关联查询商品信息（名称、价格、库存、图片）
   - 前端展示时无需再次请求商品详情

---

### 13. 商品分类缺失

**问题描述：**

- 有艺术分类（art_category, art_subcategory）
- 帖子（post）可以关联艺术子分类
- 但商品（product）没有关联任何分类
- 无法按类别浏览商品

**调整建议：**

1. **方案A：** 在 `product` 表增加 `subcategory_id` 字段，关联 `art_subcategory`
2. **方案B：** 创建独立的 `product_category` 表（如果商品分类与艺术分类不同）
3. 建议采用方案A，统一分类体系

---

### 14. 社交功能缺少通知

**问题描述：**

- 用户的帖子被点赞、评论、收藏时，作者不知道
- 缺少消息通知机制

**业务影响：**

- 用户参与感和活跃度降低
- 社区氛围不活跃

**调整建议：**

1. 创建通知表：

   ```sql
   notification (
       notification_id,
       user_id,           -- 接收通知的用户
       type,              -- 通知类型：like/comment/collection/order/badge
       related_id,        -- 关联对象ID（post_id, order_id等）
       content,           -- 通知内容
       is_read,           -- 是否已读
       created_at
   )
   ```

2. 在业务操作后发送通知：
   - 点赞帖子 → 通知帖子作者
   - 评论帖子 → 通知帖子作者
   - 订单状态变化 → 通知买家
   - 获得徽章 → 通知用户
