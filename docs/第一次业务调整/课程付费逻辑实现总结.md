### 2. 课程购买流程

#### 2.1 OrderitemRepository 扩展
新增查询方法检查用户是否已购买课程：

```java
@Query("SELECT COUNT(oi) > 0 FROM Orderitem oi " +
       "JOIN Order o ON oi.orderId = o.orderId " +
       "WHERE o.userId = :userId " +
       "AND oi.courseId = :courseId " +
       "AND o.status IN ('paid', 'shipped', 'completed')")
boolean existsPaidCourseByUserIdAndCourseId(@Param("userId") Integer userId,
                                             @Param("courseId") Integer courseId);
```

**关键点：**
- JOIN 订单表检查订单状态
- 只有已支付的订单才算购买成功
- 状态包括：`paid`（已支付）、`shipped`（已发货，虚拟商品不适用）、`completed`（已完成）

#### 2.2 OrderService.purchaseCourse() 实现
直接购买课程的完整流程：

1. **验证课程存在**
2. **检查课程类型** - 必须是付费课程
3. **防重复购买** - 检查用户是否已购买
4. **生成订单号** - 使用统一的订单号生成规则
5. **创建订单主表** - 状态为 `pending`（待支付）
6. **创建课程订单项** - 包含价格快照和课程信息快照
   - `courseId` - 课程ID
   - `productId` - NULL（课程订单不关联商品）
   - `quantity` - 1（课程数量固定为1）
   - `priceAtPurchase` - 课程价格快照
   - `productNameSnapshot` - 课程标题快照
   - `productImageSnapshot` - 课程封面快照

**特点：**
- 课程订单无需地址（虚拟商品）
- 创建后立即可支付
- 支付后即可报名学习

### 3. 课程报名流程改造

#### 3.1 CourseEnrollmentService.enrollCourse() 增强
在原有报名逻辑基础上增加付费检查：

```java
// 检查课程类型（核心逻辑）
if ("paid".equalsIgnoreCase(course.getType())) {
    // 付费课程：检查用户是否已购买
    boolean hasPurchased = orderitemRepository.existsPaidCourseByUserIdAndCourseId(userId, courseId);
    if (!hasPurchased) {
        throw new BizException(ResultCodes.INVALID_PARAM,
            "该课程为付费课程，请先购买后再报名");
    }
}
// 免费课程：直接允许报名
```

**流程说明：**
1. 检查课程是否存在
2. 检查是否重复报名
3. **核心：检查课程类型**
   - 免费课程：直接允许报名
   - 付费课程：验证是否已购买（订单已支付）
4. 创建报名记录

## 🔄 完整业务流程

### 流程1：免费课程学习

```
1. 用户浏览课程列表
   GET /course/list
   
2. 查看课程详情，发现是免费课程 (type="free")
   GET /course/{courseId}
   
3. 直接报名
   POST /course/enroll
   Body: {"courseId": 1}
   
   系统自动：
   - 检查课程类型（免费）
   - 创建报名记录
   
4. 开始学习
   POST /course/enroll/complete
   Body: {"courseId": 1, "chapterId": 1}
```

### 流程2：付费课程学习

```
1. 用户浏览课程列表
   GET /course/list
   
2. 查看课程详情，发现是付费课程 (type="paid", price=99.00)
   GET /course/{courseId}
   
3. 检查是否已购买（前端可选）
   GET /course/enroll/check-purchased/{courseId}
   返回：{"data": false}
   
4. 购买课程
   POST /order/purchase-course/{courseId}
   
   系统自动：
   - 检查课程类型（付费）
   - 防止重复购买
   - 生成订单（status=pending）
   - 创建课程订单项
   
   返回：订单信息（包含orderId和orderNumber）
   
5. 支付订单
   PUT /order/{orderId}/pay
   
   系统自动：
   - 更新订单状态为 paid
   
6. 报名课程
   POST /course/enroll
   Body: {"courseId": 1}
   
   系统自动：
   - 检查课程类型（付费）
   - 验证是否已购买（检查order_item表）
   - 创建报名记录
   
7. 开始学习
   POST /course/enroll/complete
   Body: {"courseId": 1, "chapterId": 1}
```

### 流程3：付费课程购买后取消

```
1. 用户购买课程后，支付前反悔
   PUT /order/{orderId}/cancel
   
   系统自动：
   - 验证订单状态（pending或paid）
   - 更新订单状态为 cancelled
   - 课程订单无需恢复库存
   
2. 用户需要重新购买才能报名学习
```

## 🔧 技术要点

### 1. JPQL JOIN 查询
```java
@Query("SELECT COUNT(oi) > 0 FROM Orderitem oi " +
       "JOIN Order o ON oi.orderId = o.orderId " +
       "WHERE o.userId = :userId " +
       "AND oi.courseId = :courseId " +
       "AND o.status IN ('paid', 'shipped', 'completed')")
```

**关键点：**
- JOIN 两个实体类（不是表名）
- 使用 `IN` 查询多个状态
- 返回 boolean（COUNT > 0）

### 2. 可选字段设计
```java
@Column(name="product_id")
private Integer productId;  // 可为 NULL

@Column(name="course_id")
private Integer courseId;   // 可为 NULL
```

**优势：**
- 同一表支持不同类型的订单项
- 避免创建多个表
- 查询时通过 `IS NOT NULL` 过滤

### 3. 业务逻辑分层
```
Controller → Service → Repository
   ↓           ↓           ↓
接口层      业务逻辑      数据访问
```

- Controller：参数验证、结果封装
- Service：业务规则、事务控制
- Repository：数据库操作
